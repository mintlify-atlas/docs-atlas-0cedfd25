## Introduction

Vicenta is built with Next.js 14, which provides multiple deployment options and optimizations for production environments. This guide covers the build process, production optimizations, and available deployment platforms.

## Build Process

Vicenta uses the standard Next.js build system. The build process compiles your application for production, optimizing performance and creating static assets.

### Running a Production Build

To create a production build of Vicenta, run the build command:

```bash
npm run build
```

This command executes `next build`, which performs the following optimizations:

- **Code Minification**: JavaScript and CSS are minified to reduce file sizes
- **Tree Shaking**: Unused code is removed from the bundle
- **Image Optimization**: Next.js optimizes images automatically
- **Static Generation**: Pages are pre-rendered when possible for better performance
- **Code Splitting**: Automatic code splitting for faster page loads

### Testing the Production Build Locally

After building, you can test the production build locally:

```bash
npm run build
npm start
```

The `npm start` command runs `next start`, which serves the production build on `http://localhost:3000`.

<Note>
Always test your production build locally before deploying to ensure everything works as expected.
</Note>

### TypeScript Compilation

Vicenta is built with TypeScript, which is compiled during the build process:

- Type checking ensures code quality and catches errors before deployment
- TypeScript is compiled to optimized JavaScript
- Source maps can be generated for debugging (configured in `next.config.mjs`)

### CSS Processing

Tailwind CSS classes are processed during build:

- Unused styles are purged from the final bundle
- CSS is minified and optimized
- Critical CSS is inlined for faster first paint

## Environment Variables

Before deploying, ensure you configure any necessary environment variables:

### Setting Up Environment Variables

1. Create environment-specific files locally (not committed to git):
   - `.env.local` - Local development overrides
   - `.env.production` - Production-specific variables

2. Configure variables in your deployment platform's dashboard

3. Access variables in your code using `process.env.VARIABLE_NAME`

<Warning>
Never commit sensitive credentials to your repository. Always use environment variables for API keys, database URLs, and other secrets.
</Warning>

### Required Environment Variables

Check your application code for any required environment variables before deploying. Common variables include:

- API endpoints
- Authentication keys
- Database connection strings
- Third-party service credentials

## Build Output

After running `npm run build`, Next.js generates a `.next` directory containing:

- **Static HTML files** for pre-rendered pages
- **JavaScript bundles** for client-side code
- **Server code** for API routes and server components
- **Build manifest** with metadata about the build

<Info>
The `.next` directory should never be committed to version control. It's automatically generated during deployment.
</Info>
